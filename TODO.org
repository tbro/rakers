** TODO [Research] Replace QuickJS with SpiderMonkey via ~mozjs~ crate
   Swap the JS engine from QuickJS (rquickjs) to SpiderMonkey (mozjs) while
   keeping the rest of the architecture unchanged.  SpiderMonkey is ES2024+,
   handles large bundles without stack overflow, and is the same engine Firefox
   ships.  This closes the JS compatibility gap without adding browser-level
   complexity (no layout, no GPU, no event loop).

   Tradeoffs vs current:
   - Requires a C++ toolchain and a large build (mozjs is ~500 MB compiled)
   - Binary size grows significantly; startup stays fast
   - Eliminates the ~boa~ fallback (boa exists specifically to avoid C deps)
   - ~mozjs~ crate API is lower-level than rquickjs; more boilerplate

   Like the existing ~rquickjs~ / ~boa~ split, SpiderMonkey would live behind
   a ~spidermonkey~ feature flag so consumers can still opt into the
   lightweight QuickJS build.  The three-way choice at compile time would be:
   ~boa~ (pure Rust, no C), ~rquickjs~ (default, QuickJS), ~spidermonkey~
   (full ES2024+, largest build).

   Worth prototyping on a branch once the current stub-improvement work
   plateaus.

** TODO [Research] Embed full Servo engine for pixel-perfect rendering
   Replace the QuickJS + html5ever stack with embedded Servo to get full CSS
   layout, real network, and SpiderMonkey JS — essentially a lightweight
   headless browser comparable to prerender.io.

   Blockers as of 2025-10:
   - Servo's embedder API (~servo::Servo~, ~EmbedderMethods~) is unstable and
     not published to crates.io; must depend on the git repo directly.
   - Requires SpiderMonkey, a GL/GPU surface (even headless), and an async
     event loop — substantially more complex than the current architecture.
   - Build time ~30 min; binary hundreds of MB; memory ~200 MB at runtime.
   - Would abandon the lightweight value proposition entirely.

   This is the right path only if rakers pivots from "fast scraping tool" to
   "embeddable Rust headless browser".  Track Servo embedding maturity; revisit
   when the ~servo~ crate is stable on crates.io.

** DONE Mithril (0 errors, no render)
   ~https://todomvc.com/examples/mithril/~

   Mithril 0.x renders via ~m.route()~ which reads ~window.location.hash~
   to pick a route.  When hash is empty the router falls through and nothing
   renders.

   Approach:
   - Check whether appending ~#/~ to the URL passed to rakers makes the
     Mithril hash router resolve the default route.
   - Alternatively inspect whether ~m.render(container, vnode)~ is called
     at all; if it is, the problem is in how we serialise the rendered tree.

** DONE Elm (1 error: ~cannot read property 'length' of undefined~)
   ~https://todomvc.com/examples/elm/~

   Elm's compiled kernel reads ~.length~ on something undefined in our stub
   environment.  Most likely candidate: ~classList~ is a plain object with
   no ~length~ property, or ~childNodes~ on a stub element that is ~null~
   rather than ~[]~.

   Approach:
   - Run with ~--verbose~, read the full error stack to identify which object
     is undefined.
   - Add ~length~ as a live getter to ~classList~ (returns ~_c.length~).

** DONE RiotJS (0 errors, no render)
   ~https://todomvc.com/examples/riotjs/~

   The page loads its template via ~<script type="riot/tag" src="js/todo.html">~
   (skipped by rakers) and then expects Riot's compiler to fetch it at
   runtime via XHR.  Our XHR stub returns empty ~responseText~, so the
   compiler has nothing to compile.

   Approach (preferred): make the XHR stub perform a real HTTP GET and store
   the response body in ~responseText~ before firing ~onload~.  This fixes
   Riot and benefits any framework that XHR-loads templates.  The fetch must
   respect the same ~HttpConfig~ (headers, UA) as the main page fetch.

   Completed: four fixes required:
   1. XHR relative URL resolution (resolve via ~_r_parse_url~ before ~_r_fetch_sync~)
   2. Synchronous XHR mode (~xhr.open('GET', url, false)~) — riot compiler used async
   3. ~querySelector('todo')~ for bare non-standard tag names (creates element in body)
   4. ~querySelectorAll~ compound selector splitting (comma-separated)
   5. Script injection: Riot compiler sets ~script.text = compiled~ and appends to
      ~document.documentElement~; ~appendChild~ now evals ~child.text~ when tag is SCRIPT
   6. ~firstChild~ / ~lastChild~ getters on ~_r_el~; ~innerHTML~ setter creates a raw-HTML
      proxy node so ~while(p.firstChild) M.appendChild(p.firstChild)~ can move content
   7. ~appendChild~ gains real move semantics (removes from source parent)

** DONE Lit (1 error: ~unsupported keyword: export~)
   ~https://todomvc.com/examples/lit/dist/~

   The Lit TodoMVC bundle is a native ES module (~export const~ at top level).
   Our skip heuristic correctly refuses it; no clean workaround exists without
   a full module loader.

   Approach: document the limitation in the README.  The Lit example is the
   only TodoMVC case that ships a naked ES-module bundle with no IIFE fallback.

** DONE Ember (1 error: meta config tag missing)
   ~https://todomvc.com/examples/emberjs/todomvc/dist/~

   Ember reads its environment config from a ~<meta name="todomvc/config/environment">~
   tag.  Fixed by collecting ~<meta>~ tags in Rust (~dom.rs~), exposing them via
   ~_r_meta~ in JS, and extending ~document.querySelector~ to resolve
   ~meta[name="X"]~ lookups against ~_r_meta~.

   The final blocker was Ember 5.x's ~hasDOM~ check in the webpack bundle
   (module 7513): it requires ~self.constructor === Window~.  In QuickJS the
   global object's ~constructor~ defaults to ~Object~, so ~hasDOM = false~ and
   the Glimmer renderer never initialised.  Fixed by overriding
   ~window.constructor~ to point at ~window.Window~ in bootstrap.js.

   Score: 19 / 20 pass (Lit is the only remaining failure).
